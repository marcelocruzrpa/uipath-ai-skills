# UI Automation Activities API Reference

Reference for the `uiAutomation` service from `UiPath.UIAutomation.Activities` package.

**Required package:** `"UiPath.UIAutomation.Activities": "[25.10.21]"`

**Auto-imported namespaces:** `System`, `UiPath.UIAutomationNext.API.Contracts`, `UiPath.UIAutomationNext.Enums`, `UiPath.UIAutomationNext.API.Models`

**Service accessor:** `uiAutomation` (type `IUiAutomationAppService`)

---

## Overview

The UI Automation API provides coded workflow access to UI interactions with desktop applications, web browsers, and SAP. The main entry point is `IUiAutomationAppService`, accessed via the `uiAutomation` service accessor.

### Workflow Pattern

1. **Open** or **Attach** to an application screen ‚Äî returns a `UiTargetApp` handle.
2. Use the `UiTargetApp` handle to perform element interactions (Click, TypeInto, GetText, etc.).
3. The `UiTargetApp` is `IDisposable` ‚Äî use `using` blocks or dispose manually.

### Screen Handle Affinity (Critical)

**Each `UiTargetApp` handle is bound to a specific screen.** Element descriptors can ONLY be used with the handle for the screen they belong to. Using a descriptor from Screen A on a handle attached to Screen B will fail with `"Target name 'X' is not part of the current screen."`.

```csharp
// CORRECT ‚Äî use Home elements on the homeScreen handle
var homeScreen = uiAutomation.Open(Descriptors.MyApp.Home);
homeScreen.Click(Descriptors.MyApp.Home.Products);   // OK ‚Äî Products is a Home element
homeScreen.Click(Descriptors.MyApp.Home.Loans);       // OK ‚Äî Loans is a Home element

// Then attach to the next screen for its elements
var formScreen = uiAutomation.Attach(Descriptors.MyApp.Form);
formScreen.TypeInto(Descriptors.MyApp.Form.Email, "test@example.com");  // OK

// WRONG ‚Äî using a Home element on the Form screen handle
formScreen.Click(Descriptors.MyApp.Home.Loans);  // FAILS: "Target name 'Loans' is not part of the current screen"
```

**When navigating multi-screen flows:** perform all interactions for one screen before attaching to the next. If the page changes after a click, attach to the new screen and use that handle for subsequent interactions.

### Target Resolution

Each method on `UiTargetApp` accepts targets in multiple forms:
- **`string target`** ‚Äî a target name defined in the Object Repository screen.
- **`IElementDescriptor elementDescriptor`** ‚Äî a strongly-typed Object Repository descriptor (e.g., `Descriptors.MyApp.LoginScreen.Username`).
- **`TargetAnchorableModel target`** ‚Äî accessed via the `UiTargetApp` indexer: `app["targetName"]` or `app[Descriptors.MyApp.Screen.Element]`.
- **`RuntimeTarget target`** ‚Äî a runtime target returned by `GetChildren` or `GetRuntimeTarget`.

### Finding Descriptors

**MANDATORY for any workflow that uses `uiAutomation.*` calls.** Follow this decision tree in **strict order** ‚Äî stop at the first step that yields the descriptor you need.

> **CRITICAL:** Steps 1 ‚Üí 2 ‚Üí 3 ‚Üí 4 MUST be followed sequentially. You MUST NOT skip to Step 4 (UITask) without completing Steps 1-3. Even if a form has many fields, indicate ALL of them in Step 3 rather than falling back to UITask. UITask is ONLY for genuinely brittle selectors where indication fails to produce reliable results.

---

#### Step 1 ‚Äî Check the project's Object Repository

Read `<PROJECT_DIR>\.local\.codedworkflows\ObjectRepository.cs`. This file is auto-generated by Studio and contains a `Descriptors` class with the hierarchy `Descriptors.<App>.<Screen>.<Element>`.

- If the descriptor you need is present ‚Üí **use it directly** and stop here.
- If the file is empty or the required screen/element is missing ‚Üí proceed to Step 2.

**Important: Add the ObjectRepository using statement** ‚Äî any workflow file that references `Descriptors.*` needs:
```csharp
using <ProjectNamespace>.ObjectRepository;  // e.g. using RoboticEnterpriseFramework.ObjectRepository;
```
Without this, you get `CS0103: The name 'Descriptors' does not exist in the current context`.

---

#### Step 2 ‚Äî Check UILibrary NuGet packages

**ALWAYS check** `project.json` ‚Üí `dependencies` for UI Library packages that may already contain pre-built screens and element descriptors for the target application. This step is NOT optional ‚Äî skipping it means you may miss existing descriptors and unnecessarily indicate elements or fall back to UITask.

**How to identify UILibrary packages:** Look in `project.json` ‚Üí `dependencies` for packages whose names follow the pattern `*.UILibrary`, `*.ObjectRepository`, `*.Descriptors`, or `*.UIAutomation` (e.g. `MyCompany.SalesApp.UILibrary`, `MultipleApps.Descriptors`). These are custom libraries that expose `Descriptors.*` classes for a specific application. If no such package is listed, skip to Step 3.

**Discovery approach ‚Äî inspect the UILibrary package:**

```bash
# Inspect the UI Library package listed in project.json
rpa-tool inspect-package --package-name <UILibraryPackageName> --package-version <Version>
```

Look for a `Descriptors` class or types ending in `Descriptor` / `ScreenDescriptor` / `ElementDescriptor` in the output.

The tool automatically checks the local NuGet cache (`~/.nuget/packages/`) if the package cannot be downloaded. If the package is already cached locally (e.g. from a private feed), you can also inspect the `.nupkg` file directly:

```bash
rpa-tool inspect-package --nupkg-path "$HOME/.nuget/packages/<package-name-lowercase>/<version>/<package-name-lowercase>.<version>.nupkg"
```

If the tool still cannot locate the package, **fall back to reading metadata files manually**:

```bash
find "$HOME/.nuget/packages/<package-name-lowercase>/<version>" -name ".metadata" -exec cat {} \;
```

UILibrary packages contain an `.objects/` directory with `.metadata` files that describe the App ‚Üí AppVersion ‚Üí Screen ‚Üí Element hierarchy. Read these files to discover what descriptors are available (App names, screen names, element names and their taxonomy types like Input, Button, Dropdown, Label).

To find the exact C# descriptor accessor paths (e.g. `Descriptors.UiPath_Banking_App.Form.Email`), inspect the package DLL using MetadataLoadContext or check the generated ObjectRepository.cs after Studio processes the package.

**Using descriptors from UILibrary packages:**
- The `using` statement uses the **package namespace**, NOT the project namespace: `using <PackageName>.ObjectRepository;`
  - Example: `using MultipleApps.Descriptors.ObjectRepository;` (NOT `using MyProject.ObjectRepository;`)
- App names with spaces are sanitized to use underscores: `"UiPath Banking App"` ‚Üí `Descriptors.UiPath_Banking_App`
- The hierarchy is: `Descriptors.<App>.<Screen>.<Element>`

- If the library exposes the descriptor you need ‚Üí **add the package to `project.json`** (if not already present), add the corresponding `using` statement, and use the descriptor.
- If there is no usable selector/descriptor in any library ‚Üí proceed to Step 3.

See [inspect-package-tool-guide.md](../inspect-package-tool-guide.md) for full CLI usage and examples.

---

#### Step 3 ‚Äî Indicate the element via Studio

Use the `indicate-application` / `indicate-element` commands to let the user point at the UI and capture selectors. Studio writes the result into the Object Repository and regenerates `ObjectRepository.cs`.

> **If `.objects/` is empty (no App exists yet):** Call `indicate-application` without `--parent-id` or `--parent-name` ‚Äî it will create the App + AppVersion + Screen automatically in a single step. No need to ask the user to manually create an App in Studio's Object Repository panel. After the indication completes, re-read `ObjectRepository.cs` to get the screen reference, then use `indicate-element` with `--parent-id` for individual elements.

> CLI flags use **kebab-case** (`--parent-id`, `--parent-name`), NOT camelCase.

```bash
# Indicate a screen ‚Äî when no App exists, omit --parent-id (creates App automatically):
cd "<PROJECT_DIR>" && rpa-tool indicate-application --name "Dashboard" --format json

# Indicate a screen ‚Äî when adding to an existing App, use --parent-id with the AppVersion reference:
cd "<PROJECT_DIR>" && rpa-tool indicate-application --name "Dashboard" --parent-id "r-xxxxx/yyyyy" --format json

# Indicate an element ‚Äî use --parent-id with the screen reference (from ObjectRepository.cs or indicate-application result)
# --activity-class-name is required for indicate-element
cd "<PROJECT_DIR>" && rpa-tool indicate-element --name "WorkItems" --parent-id "r-xxxxx/zzzzz" --activity-class-name "Click" --format json
```

> **`--parent-name` uses the AppVersion name** (e.g. `"1.0.0"`), NOT the App display name. Prefer `--parent-id` to avoid ambiguity. See [operations-guide.md](../operations-guide.md) for the full discovery procedure.

After indication, re-read `ObjectRepository.cs` to get the actual descriptor path ‚Äî Studio may normalize the name you requested.

**End-to-end example ‚Äî indicating a screen and element:**

Suppose you need to automate a login flow in an app called "NewApplication". The Object Repository has the app but is missing the `LoginScreen` screen and its `UsernameField` element.

```bash
# Discover the Object Repository structure
ls -R <PROJECT_DIR>/.objects/
# Output: jEwT/Gcsp/ ‚Äî jEwT is the App short ID, Gcsp is the AppVersion short ID

# Confirm the App metadata
# Read <PROJECT_DIR>/.objects/jEwT/.metadata ‚Üí "Name": "NewApplication", "Type": "App"

# Get the AppVersion reference
# Read <PROJECT_DIR>/.objects/jEwT/Gcsp/.metadata ‚Üí "Name": "1.0.0", "Reference": "0CEMSOp5E0Cg_qEt2bfuIA/Gcsptja5j0yy3BtGJHIhsQ"
# NOTE: --parent-name uses the AppVersion name ("1.0.0"), NOT the App name ("NewApplication")

# Indicate the screen (--parent-id preferred over --parent-name)
rpa-tool indicate-application \
  --name "LoginScreen" \
  --parent-id "0CEMSOp5E0Cg_qEt2bfuIA/Gcsptja5j0yy3BtGJHIhsQ" \
  --activity-class-name "UiPath.UIAutomationNext.Activities.NClick" \
  --description "The login screen of the application" \
  --project-dir "<PROJECT_DIR>" \
  --studio-dir "<STUDIO_DIR>" \
  --format json
# OR with --parent-name:
rpa-tool indicate-application \
  --name "LoginScreen" \
  --parent-name "1.0.0" \
  --activity-class-name "UiPath.UIAutomationNext.Activities.NClick" \
  --description "The login screen of the application" \
  --project-dir "<PROJECT_DIR>" \
  --studio-dir "<STUDIO_DIR>" \
  --format json
# Returns: a reference string for the newly created screen

# Indicate the element (user points at the username field)
rpa-tool indicate-element \
  --name "UsernameField" \
  --parent-id "<reference returned by indicate-application above>" \
  --activity-class-name "UiPath.UIAutomationNext.Activities.NClick" \
  --description "The username input field on the login screen" \
  --project-dir "<PROJECT_DIR>" \
  --studio-dir "<STUDIO_DIR>" \
  --format json

# Re-read ObjectRepository.cs and use Descriptors.NewApplication.LoginScreen.UsernameField in code
```

- If the indicated selectors work reliably ‚Üí **use the descriptors from `ObjectRepository.cs`** and stop here.
- If selectors are brittle or the UI is dynamic and hard to target precisely ‚Üí proceed to Step 4.

> **IMPORTANT:** Do NOT jump to Step 4 because indicating many elements "seems tedious." Even for complex forms with 10+ fields, indicate ALL missing screens and elements in Step 3. UITask is non-deterministic, slower, and should only be used when indicated selectors are genuinely unreliable (e.g., dynamic IDs that change every page load). Also, NEVER bypass the Object Repository by constructing `TargetAppModel` with raw URL/BrowserType ‚Äî always use descriptors.

#### Step 4 ‚Äî Use ScreenPlay (AI fallback, last resort only)

ScreenPlay (`UITask`) is an AI-powered agent that interprets natural-language instructions and performs UI interactions without requiring precise selectors. Use it **only** when step 3 selectors are not reliable enough ‚Äî it is slower and non-deterministic compared to Object Repository descriptors.

**UITask API** ‚Äî called on a `UiTargetApp` handle:

```csharp
// Simple ‚Äî always pass agentType explicitly; the default DOMBased is deprecated
UITaskResult UITask(string task, NUITaskAgentType agentType = NUITaskAgentType.DOMBased, NChildInteractionMode interactionMode = NChildInteractionMode.SameAsCard)

// With full options
UITaskResult UITask(UITaskOptions options)

// Async variants
Task<UITaskResult> UITaskAsync(string task, ...)
Task<UITaskResult> UITaskAsync(UITaskOptions options)
```

**UITaskOptions** ‚Äî created via `Options.UITask(task).With(...)`:

| Property | Type | Default | Description |
|---|---|---|---|
| `Task` | `string` | required | Natural-language instruction for the AI agent |
| `AgentType` | `NUITaskAgentType` | `DOMBased` | AI model to use |
| `InteractionMode` | `NChildInteractionMode` | `SameAsCard` | How the agent interacts with UI elements |
| `MaxIterations` | `int` | project default | Max iterations before giving up (0 = no limit) |
| `ClipboardMode` | `NTypeByClipboardMode` | project default | Whether the agent can use clipboard for typing |
| `IsDOMEnabled` | `bool` | project default | Whether the agent can use native DOM of the target app |
| `IsVariableSecurityEnabled` | `bool` | `true` | Prevents prompt injection via variables |
| `TraceAttachMode` | `NTraceAttachMode` | project default | When to attach trace files to the job |

**UITaskResult:**

| Property | Type | Description |
|---|---|---|
| `Status` | `string` | Execution status (`"Succeeded"` or failure) |
| `Result` | `string` | String result returned by the agent |
| `ErrorMessage` | `string?` | Error message (null if successful) |
| `HttpErrorStatusCode` | `string?` | HTTP status code for server-side errors |
| `TraceFiles` | `NUITaskTraceFiles?` | Trace files (based on ScreenPlay settings) |

**NUITaskAgentType:**

| Value | Model | DOM Support |
|---|---|---|
| `GeminiFlash25` | `gemini-2.5-flash` | Yes |
| `DOMBasedGPT41` | `gpt-4.1` | Yes |
| `DOMBasedGPT41Mini` | `gpt-4.1-mini` | Yes |
| `GPT5` | `gpt-5` | Yes |
| `GPT5Mini` | `gpt-5-mini` | Yes |
| `OpenAIOperator` | `openai-computer-use` | No |
| `AnthropicClaudeCU` | `claude-computer-use` | No |
| `DOMBased` | `gemini-2.0-flash` | Yes *(deprecated)* |

**NTraceAttachMode:** `OnFailure` ¬∑ `Always` ¬∑ `Never`

**Example:**

```csharp
var app = uiAutomation.Open(Descriptors.MyApp.Main);

// Simple call
var result = app.UITask("Click the Submit button", NUITaskAgentType.GeminiFlash25);

// With options
var result = app.UITask(Options.UITask("Fill in the form with name John Doe")
    .With(agentType: NUITaskAgentType.GPT5, maxIterations: 10, isDOMEnabled: true));

if (result.Status == "Succeeded")
    Log($"Agent completed: {result.Result}");
else
    Log($"Agent failed: {result.ErrorMessage}");
```

## Common Pitfalls

### Web Dropdowns and `SelectItem`

`SelectItem` may fail on web-based `<select>` dropdowns with `"Cannot select item. It was not found among existing items."` ‚Äî even after clicking the dropdown first to expand it. This is common with custom/styled web dropdowns that don't expose their items via standard accessibility APIs.

**Workaround:** Use `TypeInto` instead of `SelectItem` for web dropdowns. `TypeInto` on a `<select>` element types the value directly, which triggers the browser's native matching:

```csharp
// Instead of:
// formScreen.SelectItem(Descriptors.MyApp.Form.Term, "12");  // May fail on web dropdowns

// Use:
formScreen.TypeInto(Descriptors.MyApp.Form.Term, "12");  // Works reliably
```

### Screen Handle Mismatch

Using an element descriptor on the wrong screen handle causes `"Target name 'X' is not part of the current screen."`. Always ensure you use the correct handle for each screen's elements. See **Screen Handle Affinity** above.

### Options Pattern

Most operations accept an optional `*Options` parameter for advanced configuration. Use the `Options` static class for fluent builder creation:

```csharp
// Simple ‚Äî uses defaults
app.Click("myButton");

// With options
app.Click("myButton", Options.Click(NClickType.Double, NMouseButton.Left).With(keyModifiers: NKeyModifiers.Ctrl));
```

All option types inherit from `TargetOptions` which provides common properties:
- `Timeout` (double, default 30s) ‚Äî maximum time to find the element.
- `DelayAfter` (double, default 0.3s) ‚Äî delay after the action.
- `DelayBefore` (double, default 0.2s) ‚Äî delay before the action.

Use `.WithTimeouts(timeout, delayAfter, delayBefore)` on any option to override these.

Use `.WithContinueOnError(true)` on supported options to continue on error.
